# Limpeza, transformación e carga de datos en Power BI

## 1. Introdución

Unha vez importadas as fontes, o seguinte paso é preparar os datos para que poidan cargarse correctamente no modelo. Esta fase inclúe tarefas de limpeza, selección de columnas, corrección de valores, combinación de información e definición adecuada dos tipos de datos.

Este documento segue como referencia o laboratorio oficial **Clean, transform, and load data in Power BI** de Microsoft Learn, adaptándoo a un enfoque máis gradual e orientado a apuntamentos.

O obxectivo é entender que Power Query non serve só para mirar datos: serve para **construír un proceso reproducible de preparación dos datos**.

---

## 2. Que se vai traballar

Neste bloque van aparecer transformacións moi habituais en Power BI:

- renomear consultas e columnas
- filtrar filas
- eliminar columnas innecesarias
- combinar columnas
- expandir táboas relacionadas
- substituír valores incorrectos
- crear columnas personalizadas
- cambiar tipos de datos
- decidir que consultas cargan ao modelo

Ao final do proceso, os datos deberían quedar listos para pasar ao modelado.

---

## 3. Abrir Power Query para transformar os datos

Se os datos xa foron importados no documento anterior, abre Power Query mediante a opción `Transformar datos`.

[Oco para imaxe: acceso a Transformar datos desde Power BI Desktop]

Cada transformación que se aplica xera un novo paso. Isto é importante porque permite revisar, modificar ou eliminar transformacións posteriores sen ter que empezar de novo.

[Oco para imaxe: panel de consultas e Applied Steps]

## 4. Renomear consultas e deixar unha estrutura máis clara

Unha das primeiras tarefas recomendables é renomear as consultas para que resulten claras e útiles no modelo final.

No laboratorio oficial de Microsoft, por exemplo, táboas como `DimEmployee` pasan a nomes máis directos, como `Salesperson`.

Este criterio é moi útil tamén aquí: os nomes das consultas acabarán sendo os nomes das táboas no modelo, polo que convén que sexan comprensibles.

Exemplos de boas prácticas:

- nomes curtos pero expresivos
- evitar prefixos técnicos innecesarios cando non aportan valor didáctico
- usar nomes consistentes entre consultas e futuras visualizacións

[Oco para imaxe: cambio de nome dunha consulta]

---

## 5. Filtrar e seleccionar só o que interesa

Ao importar unha fonte, non sempre interesa conservar toda a información.

### 5.1. Filtrado de filas

O filtrado permite quedar só cos rexistros relevantes.

No laboratorio de Microsoft úsase, por exemplo, un filtro para conservar só as persoas que realmente son comerciais. A idea didáctica é moi importante: **filtrar non é ocultar datos, senón acoutar o conxunto ao que realmente se vai analizar**.

Exemplos de filtrado razoables:

- quedar só con rexistros válidos
- eliminar filas de control ou proba
- conservar só produtos comercializables

[Oco para imaxe: filtrado dunha columna]

### 5.2. Eliminar outras columnas

Tamén convén seleccionar só as columnas que realmente se van usar.

Isto ten varias vantaxes:

- simplifica o modelo
- reduce ruído
- fai máis fácil entender os datos
- evita arrastrar campos irrelevantes ás visualizacións

No laboratorio oficial, esta operación aparece repetidas veces e é unha das máis importantes en Power Query.

[Oco para imaxe: Choose Columns ou Remove Other Columns]

---

## 6. Reestruturar os datos

Ademais de filtrar e eliminar columnas, ás veces é necesario cambiar a forma dos datos.

### 6.1. Combinar columnas

Combinar columnas serve para crear un campo máis útil a partir de dous ou máis existentes.

No laboratorio oficial, por exemplo, combínanse nome e apelidos para construír unha columna co nome completo da persoa comercial.

Esta operación é útil cando se quere:

- mellorar a lexibilidade
- preparar un campo para informes
- evitar ter que repetir esa combinación máis adiante

[Oco para imaxe: Merge Columns]

### 6.2. Expandir columnas relacionadas

Nalgúns casos unha columna non contén un valor simple, senón un enlace a outra táboa relacionada. Power Query permite expandir esa información e traer campos concretos desa táboa.

Este tipo de transformación é moi habitual cando se quere enriquecer unha consulta sen cargar directamente toda a estrutura de orixe.

No laboratorio oficial úsase para obter subcategoría, categoría ou información xeográfica.

[Oco para imaxe: expandir unha columna relacionada]

### 6.3. Desnormalizar ou despivotar

Outra transformación moi frecuente é converter columnas repetidas nun formato máis analítico.

No laboratorio oficial, a táboa de obxectivos mensuais ten columnas `M01` a `M12`, e utilízase a operación de **Unpivot Other Columns** para convertelas nunha estrutura máis adecuada.

Esta transformación é especialmente importante porque mostra unha idea clave: **a estrutura boa para almacenar datos non sempre é a mellor para analizalos**.

[Oco para imaxe: Unpivot Other Columns]

## 7. Corrixir problemas de calidade

A limpeza tamén implica detectar e corrixir pequenos problemas nos valores.

### 7.1. Substituír valores inconsistentes

No laboratorio oficial aparece un exemplo moi claro: dúas variantes do mesmo valor textual, `Warehouse` e `Ware House`.

Este tipo de problema é moi común en datos reais. A solución máis sinxela adoita ser `Replace Values`.

[Oco para imaxe: Replace Values]

### 7.2. Promover a primeira fila como cabeceira

Nalgunhas fontes, especialmente en `CSV`, a primeira fila pode conter os nomes das columnas pero non terse interpretado así automaticamente.

Nese caso convén usar a opción para converter a primeira fila en cabeceiras.

### 7.3. Revisar nulos e erros

Tamén hai que prestar atención a:

- valores nulos
- erros de conversión
- columnas con formatos mixtos

Non sempre hai que resolvelo todo nesta fase, pero si deixar a consulta o máis consistente posible antes da carga.

---

## 8. Crear columnas calculadas en Power Query

Power Query permite crear columnas personalizadas cando é necesario derivar un valor novo a partir doutros campos.

No laboratorio oficial úsase esta técnica para xerar un custo cando o valor orixinal está baleiro, combinando a cantidade pedida co custo estándar do produto.

A idea importante aquí non é memorizar a fórmula concreta, senón entender o criterio:

- se un dato falta
- e existe información suficiente para reconstruílo
- pódese crear unha columna nova que resolva ese problema antes da carga

[Oco para imaxe: Custom Column]

Este tipo de operación pode ser moi útil, pero convén aplicala con sentido e deixar claro que problema resolve.

---

## 9. Tipos de datos e carga ao modelo

Unha vez feitas as transformacións principais, hai que revisar de novo os tipos de datos.

Este punto é especialmente importante en columnas como:

- cantidades
- prezos
- vendas
- custos
- datas

O laboratorio oficial insiste con razón nesta cuestión: escoller o tipo correcto non é un detalle menor. Afecta a cálculos, agregacións, ordenacións e análises posteriores.

En particular, para valores financeiros adoita ser recomendable empregar tipos decimais apropiados e non deixar que Power BI decida automaticamente sen revisión.

[Oco para imaxe: cambio de tipo de dato nunha columna numérica]

---

## 10. Decidir que consultas cargan ao modelo

Non todas as consultas teñen por que cargarse como táboas do modelo.

Ás veces unha consulta serve só como apoio para enriquecer outra, por exemplo nunha combinación ou nunha operación de merge. Nese caso pódese desactivar a súa carga ao modelo final.

Esta decisión axuda a:

- evitar táboas redundantes
- manter o modelo máis limpo
- reducir confusión na vista de datos e no panel de campos

[Oco para imaxe: desactivar Enable Load]

---

## 11. Pechar e aplicar

Cando as consultas xa están preparadas, hai que pechar Power Query e aplicar os cambios para cargar os datos ao modelo.

[Oco para imaxe: Close & Apply]

A partir dese momento, as táboas xa estarán dispoñibles en Power BI Desktop para o seguinte paso: o modelado.

---

## 12. Ideas clave deste bloque

Ao rematar este documento deberías quedar con varias ideas claras:

- Power Query é unha ferramenta de preparación, non só de visualización previa.
- Cada transformación queda rexistrada como un paso reproducible.
- Convén filtrar, seleccionar e limpar antes de cargar.
- Os tipos de datos son unha decisión analítica importante.
- Non todas as consultas teñen que acabar cargadas no modelo.

---

## 13. Punto de continuidade

Unha vez cargados os datos limpos e transformados, o seguinte paso será entrar no **modelado de datos**.
