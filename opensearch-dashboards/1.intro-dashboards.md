# OpenSearch Dashboards: introdución aos dashboards

## 1. Que é un dashboard de OpenSearch e como se accede a el

Un **dashboard de OpenSearch** é un panel no que se agrupan varias visualizacións para analizar datos dunha forma rápida e visual.

Nun mesmo dashboard pódense combinar:

- métricas resumo
- gráficas de liñas
- gráficas de barras
- táboas
- filtros interactivos

O obxectivo é ter nunha soa pantalla unha visión xeral dos datos e facilitar tarefas como:

- analizar información
- detectar patróns
- identificar incidencias ou anomalías
- comparar valores entre categorías ou períodos de tempo

### Como se accede

Para traballar con dashboards accédese á interface web de **OpenSearch Dashboards** desde o navegador.

O acceso habitual segue estes pasos:

1. Abrir a URL do servidor de OpenSearch Dashboards.
2. Iniciar sesión, se a instalación require autenticación.
3. Entrar no menú lateral e abrir a sección de visualizacións ou dashboards.

Segundo a versión e a configuración, os nomes das opcións poden variar lixeiramente, pero normalmente aparecen apartados como:

- `Dashboards`
- `Visualize`
- `Discover`
- `Index Management`

No noso caso, o acceso a dashboards faise desde a sección de `Dashboards`, onde se poden crear novos paneles ou editar os existentes.

## 2. Pasos previos

Antes de crear un dashboard é necesario preparar os datos para que OpenSearch poida interpretalos correctamente.

### 2.1. Crear un índice cun mapping adecuado

O primeiro paso é crear un **índice** onde se almacenarán os documentos.

Ademais, convén definir un **mapping**, é dicir, a estrutura dos campos e o tipo de dato de cada un.

Por exemplo, podemos crear un índice chamado `eventos` co seguinte mapping:

```json
PUT eventos
{
  "mappings": {
    "properties": {
      "endpoint": {
        "type": "keyword"
      },
      "ip": {
        "type": "ip"
      },
      "latency_ms": {
        "type": "integer"
      },
      "mensaxe": {
        "type": "text"
      },
      "nivel": {
        "type": "keyword"
      },
      "request_id": {
        "type": "keyword"
      },
      "servizo": {
        "type": "keyword"
      },
      "status": {
        "type": "integer"
      },
      "timestamp": {
        "type": "date"
      },
      "usuario": {
        "type": "keyword"
      }
    }
  }
}
```

Neste exemplo:

- `timestamp` úsase como campo temporal
- `servizo`, `nivel`, `usuario`, `endpoint` e `request_id` están definidos como `keyword`, o que permite filtrar e agrupar facilmente
- `latency_ms` e `status` son campos numéricos, útiles para cálculos e métricas
- `ip` está definida como tipo `ip`
- `mensaxe` é un campo de texto, pensado para buscas sobre o contido

Definir ben o mapping é importante porque determina:

- que campos se poden filtrar
- que campos se poden agregar
- que tipos de visualización son posibles

Se un campo está mal tipado, despois pode non servir para facer gráficas ou agregacións.

### 2.2. Crear un index pattern

Unha vez creado o índice, o seguinte paso é crear o **index pattern** en OpenSearch Dashboards.

O index pattern é a referencia que usa a interface para traballar cos índices desde `Discover`, as visualizacións e os dashboards.

Para crealo, accédese normalmente desde:

- `Dashboards Management -> Index patterns`

Neste caso, como o índice se chama `eventos`, o máis normal sería crear un index pattern co mesmo nome:

- `eventos`

Ao crear o index pattern debemos indicar tamén o campo temporal:

- `timestamp`

Así OpenSearch Dashboards poderá:

- ordenar os datos no tempo
- aplicar filtros por rango temporal
- construír visualizacións baseadas en evolución temporal

Se se traballase con varios índices semellantes, tamén se podería usar un patrón máis amplo, pero neste exemplo abonda con `eventos`.

Sen este paso, OpenSearch Dashboards non poderá explorar os datos nin utilizalos nas visualizacións.

### Resumo dos pasos previos

Antes de crear un dashboard debemos ter:

1. Un índice creado.
2. Un mapping correcto.
3. Un index pattern configurado.

## 3. Crear un dashboard

Un dashboard organízase a partir de visualizacións que xa existen ou que se crean durante o proceso.

### 3.1. Pasos básicos

O fluxo de traballo máis habitual é o seguinte:

1. Crear ou revisar o index pattern.
2. Crear visualizacións a partir dese index pattern.
3. Crear un dashboard novo.
4. Engadir as visualizacións ao dashboard.
5. Organizar os elementos na pantalla.
6. Gardar o dashboard.

Nalgúns casos tamén se poden engadir filtros e controis interactivos para facer o panel máis útil.

### 3.2. Conceptos básicos

Ao crear un dashboard convén ter claros varios conceptos.

#### Visualización

É cada un dos elementos gráficos que se engaden ao panel.

Exemplos:

- un KPI co número total de eventos
- unha gráfica de liñas coa evolución temporal
- unha gráfica de barras por servizo
- unha táboa con información agregada

Ademais das gráficas e táboas, tamén se pode engadir un panel de texto en formato `Markdown` ou `Text` para incluír:

- títulos
- descricións curtas
- explicacións sobre o contido do dashboard
- instrucións para o usuario

#### Métricas

Son operacións que calculan valores sobre os datos.

Exemplos habituais:

- `Count`
- `Average`
- `Min`
- `Max`
- `Percentiles`

#### Buckets

Serven para agrupar os datos.

Exemplos habituais:

- `Terms` para agrupar por categorías
- `Date Histogram` para agrupar por tempo
- `Filters` para crear subconxuntos de datos

#### Filtros

Permiten restrinxir a información que se mostra.

Poden ser:

- globais para todo o dashboard
- propios dunha visualización
- interactivos mediante controis

#### Split series

Permite mostrar varias series dentro da mesma gráfica.

Por exemplo:

- total de eventos fronte a eventos de erro
- comparación de varios servizos nun mesmo gráfico temporal

### 3.3. Estrutura recomendada dun dashboard

Un bo dashboard non debe ser só correcto tecnicamente, tamén debe ser claro e fácil de interpretar.

Unha estrutura recomendada pode ser esta:

#### Parte superior: visión xeral

Na zona superior é útil colocar indicadores rápidos:

- número total de eventos
- número total de erros
- latencia media
- porcentaxe de éxito

Esta parte permite ter un resumo inmediato da situación.

#### Parte central: evolución e comparación

Na zona central adoitan colocarse as visualizacións principais:

- gráfica de liñas para ver a evolución no tempo
- gráfica de barras para comparar categorías
- área apilada se interesa ver volume acumulado

Esta sección é a máis importante para detectar tendencias, picos e cambios de comportamento.

#### Parte inferior: detalle e apoio

Na parte inferior pódense incluír:

- táboas con agregacións
- repartición por niveis ou estados
- filtros adicionais

Aquí o usuario pode facer unha análise máis detallada do que viu na parte superior.

### 3.4. Recomendacións prácticas

Ao deseñar un dashboard convén seguir estas recomendacións:

- non saturar a pantalla con demasiadas visualizacións
- poñer primeiro a información máis relevante
- usar títulos claros en cada panel
- empregar filtros para facilitar a exploración
- manter unha distribución ordenada e coherente

O dashboard debe permitir entender rapidamente o estado xeral e, ao mesmo tempo, afondar nos detalles cando sexa necesario.
